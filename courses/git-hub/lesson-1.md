## 📦 1. `git init` — Создать сундук

![image](https://github.com/user-attachments/assets/d1eb245f-afb9-407f-b0ea-a264458304bd)

**Что делает эта команда:**

Превращает обычную папку в репозиторий. Внутри папки появляется скрытая папка `.git`, в которой Git хранит всю историю изменений. Без `git init` никакие другие команды Git не будут работать — сундука просто нет.

**Когда использовать:**

Один раз в начале проекта. Пришёл, создал папку, написал `git init` — и папка стала репозиторием.

**Как выглядит в терминале:**

```bash
mkdir my-project
cd my-project
git init
# Вывод: Initialized empty Git repository in /my-project/.git/
```

Фраза Термикса:

«Йо-хо-хо! Я создал этот пустой сундук. Теперь клади сюда свои сокровища!»

## 🔍 2. git status — Осмотреть палубу

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix смотрит в подзорную трубу]
```

Что делает эта команда:

Показывает, что изменилось с момента последнего коммита. Какие файлы новые (красные), какие добавлены в staging (зелёные), какие изменены, какие удалены. Ничего не меняет — только показывает.

Когда использовать:

Постоянно. Перед каждым коммитом. После каждого изменения. Если сомневаешься — git status. Это самая безопасная команда.

Как выглядит в терминале:

```bash
git status
# On branch main
# Untracked files: (красные — не отслеживаются)
#   README.md
# Changes to be committed: (зелёные — готовы к коммиту)
#   index.html
```

Фраза Термикса:

«Кэп, осмотрелся на палубе! Вот что мы сломали (красное), а что починили (зелёное).»

## 📦 3. git add . — Сгрести всё в кучу

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix сгребает рыбу, файлы и звёздочки в огромную кучу]
```

Что делает эта команда:

Добавляет ВСЕ изменённые и новые файлы в staging area — область подготовленных файлов. Точка . означает «всё в текущей папке». Можно добавить не всё, а конкретный файл: git add index.html.

Когда использовать:

Перед коммитом. Сделал изменения → git add . → git commit.

Как выглядит в терминале:

```bash
git add .          # добавить всё
git add index.html # добавить только один файл
git status         # проверить — файлы стали зелёными
```

Фраза Термикса:

«Хватай всё, что видишь! Тащим на палубу для инспекции. Даже ту треску, что пахнет багами!»

## 📜 4. git commit -m — Запечатать свиток

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix запечатывает свиток сургучом с серьёзным лицом]
```

Что делает эта команда:

Сохраняет снимок текущего состояния файлов в историю. Флаг -m позволяет написать сообщение прямо в команде. Сообщение должно объяснять, ЧТО сделано и ПОЧЕМУ.

Хорошие сообщения:

· git commit -m "добавил анимацию кота при клике"
· git commit -m "починил баг с исчезновением рыбы"

Плохие сообщения:

· git commit -m "fix" — что fix?
· git commit -m "update" — что update?
· git commit -m "123" — что 123?

Как выглядит в терминале:

```bash
git add .
git commit -m "создал основу сайта: HTML, CSS, JS"
# Вывод: [main abc1234] создал основу сайта: HTML, CSS, JS
#        3 files changed, 50 insertions(+)
```

Фраза Термикса:

«Запиши в бортовой журнал, зачем ты это сделал. Без записи — кину за борт!»

## 🦜 5. git push — Отправить попугая

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix отправляет зелёного попугая в небо]
```

Что делает эта команда:

Отправляет твои закоммиченные изменения с локального компьютера на GitHub (или другой удалённый сервер). После git push твой код видят другие пираты.

Когда использовать:

После каждого осмысленного коммита. Не надо пушить каждую секунду. Сделал задачу → закоммитил → запушил.

Как выглядит в терминале:

```bash
git push origin main
# origin — название удалённого репозитория (по умолчанию)
# main — ветка, в которую отправляем
```

Фраза Термикса:

«Лети, Арр-Иджин! Пусть весь мир увидит, какой ты крутой кодер!»

## 🦜 6. git pull — Принять попугая

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix ловит попугая, которого прислал другой пират]
```

Что делает эта команда:

Забирает изменения с GitHub, которые сделали другие пираты (или ты сам с другого компьютера). git pull = git fetch (забрать) + git merge (объединить с твоим кодом).

Когда использовать:

Перед началом работы. Пришёл утром → git pull → получил свежий код → работаешь.

Как выглядит в терминале:

```bash
git pull origin main
# Вывод: Updating abc1234..def5678
#        index.html | 2 ++
#        1 file changed, 2 insertions(+)
```

Фраза Термикса:

«Тяни свежий код с Гитхаба! Если порвётся конфликт — без паники, будем шить.»

## 🧬 7. git clone — Скопировать сундук

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix делает копию сундука с помощью магического луча из крюка]
```

Что делает эта команда:

Создаёт полную копию удалённого репозитория у тебя на компьютере. Копирует ВСЁ: все файлы, всю историю, все ветки. Используется один раз при первом знакомстве с проектом.

Когда использовать:

Когда хочешь поработать над чужим проектом. Или когда переустановил систему и нужно скачать свой проект с GitHub.

Как выглядит в терминале:

```bash
git clone https://github.com/username/project.git
# Вывод: Cloning into 'project'...
#        Receiving objects: 100%, done.
cd project
```

Фраза Термикса:

«Копируем чужой сундук! Только с разрешения капитана. Мы пираты, а не воры.»

## 🌴 8. git branch — Создать параллельный мир

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix раздваивается на три версии себя, как Наруто]
```

Что делает эта команда:

Создаёт новую ветку — параллельную версию кода. В ней можно экспериментировать, ломать, пробовать новое, и основная версия (main) останется нетронутой.

Когда использовать:

· Хочешь добавить новую фичу → новая ветка.
· Хочешь починить баг → новая ветка.
· Хочешь попробовать безумную идею → новая ветка.

Как выглядит в терминале:

```bash
git branch laser-cat       # создать ветку
git branch                 # посмотреть все ветки
# * main
#   laser-cat
```

Фраза Термикса:

«Отпочковался! Создал свою маленькую реальность, где ломать можно без страха.»

## 🌴 9. git switch — Прыгнуть в другой мир

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix прыгает в портал, ведущий в другую ветку]
```

Что делает эта команда:

Переключает тебя между ветками. Был в main → git switch laser-cat → ты в ветке laser-cat. Файлы на компьютере мгновенно меняются.

Когда использовать:

Когда нужно перейти работать в другую ветку.

Как выглядит в терминале:

```bash
git switch laser-cat       # переключиться на ветку laser-cat
# Switched to branch 'laser-cat'

git switch -c new-feature  # создать И переключиться (два в одном)
# Switched to a new branch 'new-feature'

git switch main            # вернуться обратно
```

Фраза Термикса:

«Прыгаю в другую реальность! Сейчас мы в мире лазерных котов. Тут всё можно!»

## ⚔️ 10. git merge — Объединить миры

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Два Термикса жмут лапы, их ветки сливаются в одну]
```

Что делает эта команда:

Сливает изменения из одной ветки в другую. Если два пирата меняли РАЗНЫЕ файлы — всё сольётся автоматически. Если меняли ОДИН И ТОТ ЖЕ файл в одном месте — будет конфликт, и Git попросит тебя рассудить.

Когда использовать:

Когда фича готова и протестирована → сливаем в main. Когда нужно забрать изменения из main в свою ветку.

Как выглядит в терминале:

```bash
git switch main            # переключаемся на main
git merge laser-cat        # сливаем laser-cat в main
# Вывод: Updating abc1234..def5678
#        laser-eye.html | 42 ++++++++++
```

Что такое конфликт:

Если два человека изменили одну и ту же строчку, Git не знает, чья версия правильная. Он покажет:

```
<<<<<<< HEAD
Корабль: Чёрный Кот
=======
Корабль: Весёлый Пёс
>>>>>>> laser-cat
```

Ты выбираешь нужное, удаляешь маркеры, сохраняешь — конфликт решён.

Фраза Термикса:

«Сливаем ветки воедино! Да прибудет с тобой сила резолвить конфликты, юнга.»

## 📖 11. git log — Полистать бортовой журнал

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix в очках профессора читает древний свиток с историей коммитов]
```

Что делает эта команда:

Показывает историю всех коммитов в обратном порядке (сначала новые). Видно: кто сделал коммит, когда, и с каким сообщением. У каждого коммита есть уникальный хеш (длинный набор букв и цифр), по которому его можно найти.

Когда использовать:

· Хочешь вспомнить, что делал на прошлой неделе.
· Нужно найти, в каком коммите сломалась фича.
· Хочешь посмотреть, кто что менял.

Как выглядит в терминале:

```bash
git log                    # полная история
git log --oneline          # краткая версия (одна строка на коммит)
# abc1234 добавил анимацию кота
# def5678 починил баг с рыбой
# 7890123 создал основу сайта
```

Фраза Термикса:

«Бортовой журнал. Тут всё: победы, ошибки, и тот раз, когда ты закоммитил пароль от сервера. Не делай так больше.»

## ⏪ 12. git revert — Отменить заклинание

```
🖼️ [МЕСТО ДЛЯ КАРТИНКИ: Termix с песочными часами, время течёт назад, баг исчезает]
```

Что делает эта команда:

Отменяет изменения конкретного коммита, создавая НОВЫЙ коммит с отменой. История не стирается — добавляется запись «вот этот коммит я отменил». Это безопасный способ исправить ошибку.

Когда использовать:

· Закоммитил баг → git revert ХЕШ_БАЖНОГО_КОММИТА → баг исчез.
· Случайно запушил не то → git revert → запушил отмену → всё чисто.

Как выглядит в терминале:

```bash
git log --oneline          # находим хеш плохого коммита
# abc1234 добавил страшный баг
# def5678 хороший коммит

git revert abc1234         # отменяем плохой коммит
# Открывается редактор — пишем сообщение, сохраняем, выходим
# Или: git revert abc1234 --no-edit (без редактирования сообщения)
```

Фраза Термикса:

«Ошибка? Не страшно. Отматываем время назад. Баг исчез, а история осталась чистой.»
