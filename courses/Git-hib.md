🏴‍☠️ КУРС «GIT ДЛЯ ЮНЫХ ПИРАТОВ»: ОТ init ДО КОЛЛАБОРАЦИИ

Миссия курса:

Превратить Git из страшного чёрного терминала с ошибками в понятную машину времени, где каждая команда — это пиратское действие, а каждый коммит — запись в бортовом журнале.

Кому подходит:

· Детям 11–14 лет, которые уже начали писать код и хотят сохранять его красиво.
· Тем, кто потерял курсач из-за того, что не нажал «Сохранить».
· Тем, кто хочет участвовать в командных проектах (игра Skitter, платформа Академии).
· Новичкам, которые слышали слова «GitHub», но думают, что это соцсеть для гиков.

Главное правило курса:

Git — это не страшно. Это твой цифровой пиратский сундук.
Не знаешь, что делает команда? Спроси у Термикса.
Боишься сломать? Мы всё починим через git reflog.

Формат обучения:

· 8 занятий по 90 минут.
· Каждое занятие: 15 минут теории через пиратскую легенду, 45 минут практики в терминале, 30 минут игры или квиза.
· После каждого занятия — мем-шпаргалка и бадж.

⚓ НЕДЕЛЯ 1: ПИРАТСКИЙ СУНДУК

Цель: Понять, зачем нужен Git, установить его и создать первый репозиторий.

Легенда:
Представь, что ты пишешь книгу. Ты написал главу 1, всё отлично. Потом написал главу 2, но что-то пошло не так. Ты удаляешь кусок текста, сохраняешь... а через час понимаешь, что удалённый кусок был гениальным. Но его больше нет.

Git — это машина времени. Каждый раз, когда ты делаешь git commit, ты как будто ставишь закладку в книге с пометкой «вот тут всё было хорошо». Ты можешь в любой момент вернуться к этой закладке, даже если потом всё сломал.

Термикс называет Git «Пиратским Сундуком». Ты кладёшь в сундук сокровища (свой код), закрываешь крышку (commit) и в любой момент можешь открыть сундук и достать то, что положил.

Практика:

1. Устанавливаем Git (Windows: git-scm.com, Mac: brew install git, Linux: sudo apt install git).
2. Открываем терминал. Термикс говорит: «Не бойся чёрного экрана, юнга. Это твой штурвал».
3. Создаём папку проекта: mkdir pirate-treasure && cd pirate-treasure.
4. Превращаем папку в репозиторий: git init.
5. Смотрим, что изменилось: git status.
6. Создаём файл README.md с текстом «Сокровища юнги [Имя]».
7. Снова git status — видим неотслеживаемый файл.

Мем-правило: git status — это твоя подзорная труба. Смотри в неё почаще, чтобы знать, что творится на палубе.

Игровая механика:
Игра «Найди сокровище». Наставник создаёт репозиторий с 5 файлами. Один из них содержит «золотую рыбку» (секретную фразу). Дети делают git init, копируют файлы в свой репозиторий и через git status находят, какие файлы изменились.

Мем-шпаргалка:
Изображение Термикса, открывающего сундук с зелёным свечением.

Команды:
git init — Создать новый сундук (репозиторий).
git status — Посмотреть, что в сундуке изменилось.

Бадж: ⚓ Создатель Сундуков.

📦 НЕДЕЛЯ 2: ГРУЗИМ ДОБЫЧУ

Цель: Научиться добавлять файлы в индекс и делать осмысленные коммиты.

Легенда:
На пиратском корабле есть трюм. Ты не можешь просто кинуть добычу куда попало — бардак погубит судно. Сначала ты складываешь добычу на палубу (staging area / индекс), проверяешь её, а потом убираешь в трюм с биркой «Вот что я положил и зачем».

· Палуба = staging area (git add).
· Трюм = репозиторий (git commit).
· Бирка = сообщение коммита (-m "описание").

Термикс орёт с бочки: «Не пиши "fix" или "update" в сообщении к коммиту! Пиши, что именно ты сделал, чтобы через месяц не гадать, куда дел пушку».

Как писать хорошие сообщения к коммитам (Пиратский ГОСТ):

Плохо: fix.
Хорошо: починил баг с исчезновением рыбы при клике.
Плохо: update.
Хорошо: добавил анимацию кота при нажатии.
Плохо: 123.
Хорошо: изменил цвет кнопки на зелёный неоновый.

Практика:

1. Создаём 3 файла: index.html, style.css, script.js.
2. Добавляем один файл: git add index.html.
3. git status — видим, что один файл в staging (зелёный), два нет (красные).
4. Добавляем все файлы: git add ..
5. Делаем коммит: git commit -m "создал основу сайта: HTML, CSS, JS".
6. Меняем файлы, снова git status, git add, git commit.

Мем-правило: git add . — Термикс хватает всю рыбу в охапку. git commit — Термикс запечатывает сургучом свиток.

Игровая механика:
Игра «Испорченный коммит». Наставник показывает 5 коммитов с плохими сообщениями. Дети должны переписать их по-человечески. Лучший вариант получает звание «Летописец».

Мем-шпаргалка:
Изображение Термикса, запечатывающего свиток с сургучом.

Команды:
git add . — Добавить всё на палубу.
git add имя_файла — Добавить конкретный файл.
git commit -m "сообщение" — Убрать в трюм с биркой.

Бадж: 📦 Мастер Коммитов.

🦜 НЕДЕЛЯ 3: ОТПРАВЛЯЕМ ПОПУГАЯ

Цель: Понять разницу между локальным и удалённым репозиторием, связать Git с GitHub, сделать push и pull.

Легенда:
Твой сундук с сокровищами пока что стоит в твоей каюте. Никто, кроме тебя, его не видит. Но пираты — командные ребята. Мы должны отправить карту сокровищ в главный штаб, чтобы другие пираты (или твой наставник Артём) могли их увидеть и помочь.

GitHub — это главный пиратский штаб в облаках.

· git push — ты отправляешь почтового попугая с копией карты в штаб.
· git pull — ты просишь попугая принести свежие карты от других пиратов.
· git clone — ты находишь чужой сундук и делаешь его копию у себя в каюте.

Термикс напутствует: «Попугай летит только в одну сторону за раз. Отправляй его часто, чтобы не потерять данные при кораблекрушении».

Практика:

1. Регистрируемся на GitHub (если ещё нет).
2. Создаём новый репозиторий на GitHub (без README, пустой).
3. Связываем локальный репозиторий с удалённым:
   git remote add origin https://github.com/ТВОЙ_НИК/pirate-treasure.git.
4. Отправляем код: git push -u origin main.
5. Проверяем на GitHub — файлы появились.
6. Меняем что-то на GitHub прямо в браузере (создаём файл map.txt).
7. Тянем изменения: git pull origin main.
8. Файл map.txt появился локально.

Разбираем ошибки:

Ошибка: failed to push some refs.
Перевод Термикса: «Кто-то уже отправил попугая в штаб раньше тебя. Сначала прими его посылку: git pull, потом отправляй свою».

Ошибка: fatal: not a git repository.
Перевод Термикса: «Ты не в той папке, юнга. Ты в открытом море, а сундука тут нет. Сделай cd в папку с проектом».

Игровая механика:
Игра «Испорченный Попугай». Наставник делит детей на пары. Ребёнок А пишет текст в файле и пушит. Ребёнок Б пуллит, меняет текст и пушит обратно. Ребёнок А пуллит. Кто первый соберёт 3 версии файла — победил.

Мем-шпаргалка:
Изображение Термикса, отправляющего зелёного попугая в небо (наша легендарная картинка из стикерпака).

Команды:
git push origin main — Отправить попугая в штаб.
git pull origin main — Принять попугая из штаба.
git clone URL — Скопировать чужой сундук.

Бадж: 🦜 Попугай-Почтальон.

🌴 НЕДЕЛЯ 4: ОСТРОВ ВЕТОК

Цель: Понять, зачем нужны ветки, и научиться их создавать и переключаться.

Легенда:
Представь, что твой код — это карта сокровищ. Ты плывёшь по основному маршруту (ветка main). Но вдруг ты думаешь: «А что, если доплыть до Острова Драконов и поискать там?». Ты не хочешь портить основную карту, поэтому рисуешь новую ветку (branch) от основного маршрута.

Если на Острове Драконов ты нашёл золото — ты переносишь находку на основную карту (merge).
Если на Острове Драконов тебя съели — ты просто выбрасываешь эту ветку и плывёшь дальше по основной.

Термикс комментирует: «Ветки — это твоя возможность экспериментировать, не ломая работающий код. Хочешь добавить лазерную пушку коту? Делай ветку laser-cat и пробуй. Если кот взорвался — удалил ветку, и основная версия цела».

Практика:

1. Смотрим, на какой мы ветке: git branch.
2. Создаём новую ветку: git branch pirate-island.
3. Переключаемся на неё: git switch pirate-island.
4. Создаём файл treasure-map.txt, коммитим.
5. Переключаемся обратно на main: git switch main.
6. Смотрим — файла treasure-map.txt нет! Магия!
7. Переключаемся обратно на pirate-island — файл снова здесь.
8. Хотим и создать, и переключиться сразу: git switch -c dragon-island.

Игровая механика:
Квест «Три Острова». Наставник даёт задание: создать 3 ветки (island-gold, island-silver, island-bronze), в каждой создать файл с описанием сокровища. Потом переключиться на main и убедиться, что файлов нет. Кто быстрее создал все 3 ветки с правильными файлами — получает звание «Картограф».

Мем-шпаргалка:
Изображение Термикса, который раздвоился (как Наруто) на три версии себя.

Команды:
git branch — Посмотреть список веток.
git branch имя — Создать ветку.
git switch имя — Переключиться на ветку.
git switch -c имя — Создать и сразу переключиться.

Бадж: 🌴 Картограф Веток.

⚔️ НЕДЕЛЯ 5: СЛИЯНИЕ ЭКИПАЖЕЙ

Цель: Научиться сливать ветки и решать конфликты без паники.

Легенда:
Два пиратских корабля плыли разными маршрутами и нашли разные сокровища. Теперь они встречаются в порту и хотят объединить добычу в один сундук. Если сокровища лежат в разных углах сундука — всё хорошо, они просто складываются вместе. Но если оба пирата положили золотой дублон в одно и то же место — начинается спор. Кто прав? Чей дублон оставить? Это и есть КОНФЛИКТ.

Термикс орёт в рупор: «Конфликт — это не ошибка! Это Git говорит: "Эй, капитан, тут два пирата претендуют на одну строчку кода. Рассуди". Ты — капитан. Ты решаешь».

Практика:

1. Создаём две ветки от main: team-kot и team-pes.
2. В team-kot пишем в ship-name.txt: «Корабль: Чёрный Кот».
3. В team-pes пишем в ship-name.txt: «Корабль: Весёлый Пёс».
4. Мержим team-kot в main: git switch main && git merge team-kot. Всё ок.
5. Мержим team-pes в main: git merge team-pes. КОНФЛИКТ!
6. Открываем ship-name.txt в редакторе и видим:
   ```
   <<<<<<< HEAD
   Корабль: Чёрный Кот
   =======
   Корабль: Весёлый Пёс
   >>>>>>> team-pes
   ```
7. Решаем конфликт: выбираем «Чёрный Кот» (или пишем «Чёрный Кот и Весёлый Пёс»).
8. Удаляем маркеры конфликта (<<<<<<<, =======, >>>>>>>), сохраняем.
9. Делаем git add ship-name.txt && git commit -m "разрешил конфликт названий кораблей".

Игровая механика:
Симуляция «Война Экипажей». Наставник заранее готовит репозиторий с 3 конфликтами в разных файлах. Дети клонируют его, мержат конфликтную ветку и пытаются разрешить все конфликты быстрее всех. Наставник проверяет, правильно ли разрешены конфликты.

Мем-шпаргалка:
Изображение двух Термиксов (зелёного и фиолетового), скрестивших сабли над одним файлом.

Команды:
git merge имя_ветки — Слить ветку в текущую.
Маркеры конфликта: <<<<<<< (текущая), ======= (разделитель), >>>>>>> (чужая).
Действие: Выбрать нужное, удалить маркеры, сохранить, add, commit.

Бадж: ⚔️ Мастер Дуэлей.

⏪ НЕДЕЛЯ 6: МАШИНА ВРЕМЕНИ

Цель: Научиться смотреть историю и отменять изменения.

Легенда:
Ты плывёшь по морю кода уже неделю. И вдруг понимаешь, что вчерашний коммит всё сломал. Кот перестал мяукать, а корабль плывёт задом наперёд. Что делать?

У Git есть бортовой журнал (git log) и машина времени (git reset, git revert).

· git log — полистать журнал и найти тот коммит, где всё было хорошо.
· git diff — посмотреть, что именно изменилось между коммитами.
· git reset — откатиться назад, стерев неудачные коммиты (как будто их и не было). Опасно!
· git revert — создать новый коммит, который отменяет изменения старого. Безопасно!

Термикс предупреждает: «git reset — это как выбросить сундук за борт и притвориться, что его никогда не было. git revert — это как положить в сундук записку: "Вот этот дублон был фальшивый, я вернул настоящий". Для новичков — только revert!».

Практика:

1. Создаём 3 коммита с разными файлами: file1.txt, file2.txt, file3.txt.
2. Смотрим историю: git log --oneline. Видим 3 коммита с хешами.
3. Смотрим разницу между последним и предпоследним: git diff HEAD~1 HEAD.
4. Отменяем последний коммит (безопасно): git revert HEAD.
5. Смотрим историю снова — появился новый коммит «Revert ...».
6. Откатываемся на 2 коммита назад (жёстко, в учебных целях): git reset --hard HEAD~2.
7. Файлы file2.txt и file3.txt исчезли!
8. Но мы можем вернуться: git reflog показывает все действия. Находим хеш и делаем git reset --hard ХЕШ.
9. Всё вернулось. Магия!

Игровая механика:
Квест «Спаси Кота». Наставник даёт репозиторий, где в одном из 5 коммитов Термикс «сломался» (изменился цвет, пропал файл, появилась ошибка). Дети должны с помощью git log и git diff найти «плохой» коммит и отменить его через git revert. Кто быстрее нашёл и починил — получает звание «Спасатель».

Мем-шпаргалка:
Изображение Термикса с песочными часами и в очках профессора.

Команды:
git log --oneline — Краткая история коммитов.
git diff — Что изменилось.
git revert ХЕШ — Безопасно отменить коммит.
git reset --hard ХЕШ — Жёсткий откат (ОПАСНО!).
git reflog — Журнал ВСЕХ действий (скорая помощь).

Бадж: ⏪ Хранитель Времени.

🏴‍☠️ НЕДЕЛЯ 7: КОМАНДА ПИРАТОВ

Цель: Научиться работать в команде через GitHub Issues и Pull Requests.

Легенда:
Одинокий пират может ограбить рыбацкую лодку. Но чтобы захватить испанский галеон, нужна команда. В программировании так же: большие проекты делают в команде. GitHub даёт нам инструменты для этого.

· Issue — это записка на доске объявлений: «Нужно починить пушку на баке», «Добавить анимацию попугая».
· Pull Request (PR) — это когда ты сделал работу в своей ветке и просишь капитана: «Прими мои изменения в основной код».
· Code Review — капитан (или другой пират) смотрит твой код и говорит: «Вот тут молодец, а вот тут убери лишнюю строчку».

Термикс наставляет: «PR — это не экзамен. Это просьба помощи. Ты говоришь: "Я сделал вот это, проверьте, всё ли правильно, и добавьте в общий проект"».

Практика:

1. Наставник создаёт общий репозиторий pirate-fleet.
2. Каждый ребёнок делает fork (копию на свой аккаунт).
3. Клонирует fork к себе: git clone URL_ФОРКА.
4. Создаёт ветку: git switch -c feature/МОЙ_НИК.
5. Вносит изменения: добавляет файл со своим именем и описанием «роли на корабле».
6. Пушит ветку в свой fork: git push origin feature/МОЙ_НИК.
7. Заходит на GitHub и создаёт Pull Request из своей ветки в репозиторий наставника.
8. Наставник делает Code Review, оставляет комментарии.
9. Ребёнок исправляет замечания, пушит изменения, PR обновляется автоматически.
10. Наставник принимает PR: «Вливаю в основной код!».

Игровая механика:
Ролевая игра «Пиратский Совет». Один ребёнок — «капитан» (делает ревью), остальные — «матросы» (отправляют PR). Каждый матрос должен добавить в общий проект свой файл с описанием улучшения для корабля. Капитан принимает или отклоняет PR с объяснением. Потом меняются ролями.

Мем-шпаргалка:
Изображение Термикса в капитанской треуголке, подписывающего свиток, а вокруг — несколько котов-матросов.

Понятия:
Fork — своя копия чужого репозитория.
Issue — задача на доске.
Pull Request — запрос на добавление твоего кода.
Code Review — проверка кода перед вливанием.

Бадж: 🏴‍☠️ Командный Игрок.

🎮 НЕДЕЛЯ 8: БИТВА КАПИТАНОВ (ФИНАЛ)

Цель: Закрепить все навыки в игровой форме.

Формат:
Git-квест из 7 станций. Каждая станция — задание на одну из тем курса. Дети проходят станции в своём темпе (или на время). За каждую станцию — баллы. Победитель получает главный бадж курса.

Станции:

1. «Создай Сундук»: git init, создать репозиторий, добавить README.
2. «Загрузи Добычу»: сделать 3 коммита с осмысленными сообщениями.
3. «Отправь Попугая»: создать репо на GitHub, связать, запушить.
4. «Нарисуй Карту»: создать 3 ветки с разными названиями, переключиться на main.
5. «Разреши Спор»: разрешить конфликт слияния в предоставленном репозитории.
6. «Отмотай Время»: найти и отменить плохой коммит.
7. «Отправь PR»: сделать fork, изменения и отправить Pull Request в репо наставника.

Подсчёт очков:

· Правильное выполнение: 10 баллов за станцию.
· Скорость: +5 баллов первому, +3 второму, +1 третьему.
· Красота сообщений коммитов: +3 балла.
· Помощь другому пирату: +5 баллов (командный дух!).

Награды:

· 70 баллов: бадж 🎮 Git-Капитан.
· 50–69 баллов: бадж 🥈 Git-Штурман.
· 30–49 баллов: бадж 🥉 Git-Боцман.
· Специальная награда «Самый смешной коммит»: стикер Termix'а.
